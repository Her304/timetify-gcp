# CLAUDE.md

**Frontend:** `npm run dev` (port 5173), `npm run build`, `npm run lint`
**Backend:** `python manage.py runserver` (port 8000), `migrate`, `makemigrations`

## Architecture

**Layout:** `frontend/` (React+Vite+Tailwind), `backend/` (Django); single `main/` app
**State:** All in `App.jsx`, passed as props. React Router v6 with `ProtectedRoute`.
**Auth:** JWT in localStorage, auto-refresh via `authenticatedFetch`. Password reset via Django's `PasswordResetView` at `/password_reset/`.
**Course upload:** `add.jsx` → `/api/courses/analyze/` → GPT-4o-mini parsing → `/api/courses/finalize/` persists. **Constraints:** 5MB max, `.pdf`/`.docx` only, magic-byte check, server-generated filename. Keep all four checks if editing; `pdf.extract_text` raises on unknown extensions.
**Home page (`/`):** `home/week_view.jsx` — Mon–Sun calendar grid (left) + side rail (right). Header reads `WEEK OF <date>` mono label + `ur week, ur ppl` Bricolage title + a live coral `<DAY> · h:mm AM` clock pill (updates every 30s via `setInterval`) + lighter `N friends live now` pill + `+ add class` dark pill → `/Add`. Day columns use a stable palette keyed by hash of `course_id`; current day column gets a coral tint; live class blocks get a coral ring; the "now" line is a coral horizontal bar. Hour range auto-clamps to data (default 8–18, expands as needed). **Class blocks** are full-width clickable buttons (no 60/40 width split) — opening `CourseDetailsModal` with that single class; if height < 56px they switch to a vertical stack (course code on top, avatars on bottom) to keep all three rows readable. **Overlap chip:** when multiple classes overlap, the primary block renders a coral-outlined `+N` chip in its top-right corner (stops propagation); clicking it opens `CourseDetailsModal` in master-detail mode with the full cluster on the left and the selected class's details on the right. Side rail: dark **LIVE NOW** card with coral `Blob`, white **TODAY** card (mono `TODAY · DAY DATE` + class count, `HAPPENING NOW` coral sub-card with time range + course id + name + location, then `COMING UP` timeline with pin-and-connector), then white **REMINDERS FOR TODAY** card listing today's assignments (lime `DUE` chip) and exams (coral `EXAM` chip) — empty state `all clear today ✓`. Reminders read from `/api/courses/` fetched once on mount inside `WeekView`. `home/snap.jsx` is **dead code** since the snap block moved to `/feed`.

**Course details modal (`home/CourseDetailsModal.jsx`):** Lazy-fetches `/api/courses/` on open, keys by `course_id`. Two modes — single class (header banner only) or cluster master-detail (left rail of avatar+code+room rows, right pane is the selected entry). The right pane always shows: colored banner (course id + name + `DAY · h:mm–h:mm · location`) and a `who's in this` avatar-chip row. Sections below (`this week`, `upcoming` exams/assignments, `full course page →` link) **only render when the course has data** — if the matched Course has empty `weeks`/`exams`/`assignments` and `has_ai_content=false`, those sections are hidden entirely. Do not reintroduce a loading skeleton or `no extra details` empty state — they were removed because the matched course often has nothing extra and the placeholder was noise.

**Feed page (`/feed`):** `feed/feed.jsx` — replaces the old home-page snap block. Header reads `ur feed` + `THU · MAY 21` mono date (uppercase, `·` separator). **Pills are sort-priorities, NOT filters** — every accepted friend always shows in both the avatar row and inbox; the pill just decides who's surfaced first (`today` → snapped-today first, `my_classes` → shared-course friends first, `friends` → snap-havers first). The pill row also holds a circular peach `N on` badge counting live snap-havers (last 30 min). **No** sort dropdown and **no** snap-card list — both were dropped along with the "showing N snaps" line; snaps are consumed only via the avatar row. **Avatar row:** `me + plus` tile first, then friends ordered by primary/rest where snap-havers get the coral ring and non-snap friends get a faded `ink-15` ring at 60% opacity; the rest-tail uses a session-stable seeded shuffle (`useState(() => Math.random())`). Click a snap-haver → `SnapViewerModal`; click a no-snap friend → `openChat(friend)` (create-or-get DM, then `navigate('/chat/<id>')`). **Me tile status:** derives a context class from `personalSchedule` (currently-happening, else next-up today) and renders class-aware mono text under the avatar — `snap for <code>` (coral) if I haven't snapped yet, or two stacked lines `snapped Xm ago` / `for <code>` if I have. Falls back to plain `u` when there's no class today. **Chat inbox** below the avatar row fetches `/api/chats/` on mount + on window focus into `chatsByFriendId` and merges with `orderedTiles`; rows render `avatar · username · preview · time chip` with a coral unread badge (`N` / `9+`) when `unread_count > 0`. Preview = last_message.content, or italic `[message removed]` when `is_removed`, or `snapped Xm ago` / `tap to chat` fallback. Always shows **all** friends regardless of pill. **Inbox sort:** unread → most recent `last_message.created_at` → snap-havers → alphabetical. Row click + no-snap avatar tile click share the same `openChat` handler (POST `/api/chats/` with `{friend_id}` — friend-gated dedup); inline spinner while in flight. **Data shape gotcha:** `friendsList` items are friendship rows (`{id, user, friend, status, friend_details}`), NOT user objects — flatten via `friendsList.map(f => f.friend_details).filter(Boolean)`. Server already filters to `status=1` (accepted). Camera/picker/viewer wiring matches the old snap block (course-picker modal when >1 own course). Do **not** reintroduce the old "messages coming soon" centered sheet — it was deleted when the real `/chat` route landed.

**Chat (`/chat/:roomId`):** `chat/ChatThread.jsx` — protected, 2-column layout on desktop (`md:flex-row gap-4`, `maxWidth: 1100`); mobile stacks the side panel below the chat column. **Chat column** (`md:max-w-[640px]`) is height-capped at `calc(100dvh - 120px)` so the message list and input bar stay confined while the aside scrolls independently on desktop. Header: back chevron → `/feed`, hash-colored `Avatar`, lowercase Bricolage username, mono `active Xm ago` / `last seen H:MM am` from `other_user.last_seen`. **Message list** uses `flex flex-col-reverse` so the local `messages` array stays in DESC order (matches API response shape) while rendering ascending — new messages prepend to index 0 and naturally land at the visual bottom with no manual scroll. Bubble: coral with white text for me, `T.ink08` for the other user, italic `[message removed]` at `T.ink40` for soft-deleted, dimmed 0.6 while `__pending`, `↻ retry` link when `__failed`. Per-message mono timestamp under the bubble for the newest message and any message where the gap to the next-older one is > 5 min. **Input bar**: autosizing textarea (1–5 lines), Enter sends / Shift+Enter newline, 2000-char hard cap (client clamps, server rejects 400), `chars left` counter at ≥1800 (turns coral within 50 of cap), inline paper-plane SVG (no `send` icon in brand atoms). Optimistic send appends a `pending-<ts>` row, replaced on 201 with the real row, flipped to `__failed` on error. **Load older** uses an `IntersectionObserver` on a sentinel placed last in DOM (= visual top under col-reverse) → `GET /api/chats/<id>/messages/?before=<oldest_id>`, merges by id, sets `olderExhausted` on empty response. **Polling**: 5 s `setInterval` while `document.visibilityState === 'visible'`, paused/resumed on `visibilitychange` + `focus`; merges by id so paginated older messages stay put. `markRead()` (POST `/read/`) fires on mount and whenever the poll observes a new message from the other user. Empty state: "say hi to @username 👋". Error states for `forbidden` / `not_found` / `network`. Both the mobile floating bottom-nav pill **and** the mobile top bar are **hidden on `/chat/<id>`** via a `location.pathname.startsWith('/chat/')` check in `AppShell` so neither covers the chat's own header/input. **Side panel** (right aside, ~`md:w-[320px]`) renders two cards from props threaded through the Route: `SharedClassesCard` intersects `allClasses` by `base_course` for today using my entry's `start_time/end_time/location` for display, highlighting the currently-in-session class with a coral fill + "now" chip (pure decor — no click handler); `TheirSnapsCard` filters `snapsByCourse` by `uploader_username === other_user.username` and `created_at >= startOfDay`, de-dupes by snap id, renders a 3-col 4:5 thumbnail grid (img for photos, muted/preload="metadata" video for video snaps, with `course_code` overlay chip), and clicking a thumb opens `SnapViewerModal` with `snaps={theirSnapsToday}` and `courseLabel='@username'`. Thumbnail URLs go through a local `resolveMediaUrl` helper mirroring the viewer's — relative `/media/...` paths get prefixed with `VITE_API_URL` in local dev; absolute GCS URLs pass through.

**Snap pipeline:** Add tile → 0 own courses noop, 1 straight to capture, >1 course-picker modal → `snap/SnapCaptureModal.jsx` (dark `T.ink` shell, 78px coral shutter, focus brackets, course pill on viewfinder; header title is dynamic: `snap <course code>` if current time falls inside the picked course's window via `isLiveNow`, else `snap now`; in-browser camera, `getUserMedia` 1080×1350, photo via canvas JPEG q=0.85, video via `MediaRecorder` 5s hard cap; mp4 on Safari, webm on Chrome — backend accepts both) → `POST /api/snaps/` (multipart). **Camera release:** parent's `onClose` runs `window.location.reload()` after `setCaptureCourse(null)` — relying on the React unmount cleanup alone left Chrome's camera indicator stuck. Feed: `GET /api/snaps/feed/` returns `snaps_by_course` keyed by Course PK; each snap now also includes `view_count` (count of `SnapAudience` rows with `has_viewed=True`; uploaders aren't counted because no self-audience row is written). Mark viewed: `POST /api/snaps/<id>/view/`. Uploader soft-delete: `DELETE /api/snaps/<id>/`. **Viewer (`SnapViewerModal`):** dark `T.ink` shell; receives `prevTile`/`nextTile`/`onSelectPrev`/`onSelectNext`/`onAdd` from the parent (parent tracks `viewerTileIdx` into `tiles`); renders side `Avatar`s absolutely-positioned on the backdrop — left = previous uploader (or the add icon if at the first tile, which closes the viewer and runs `handleAddClick`), right = next uploader. Internal chev-icon arrows still page within one uploader's snaps; `useEffect([snaps])` resets `idx` to 0 when switching uploaders. Header subline appends `· snap from <course>` when the snap's `created_at` HH:MM is inside that course's class window. **Constraints:** 20MB max, allowed types `image/jpeg|png` + `video/mp4|quicktime|webm`, magic-byte check, server-generated path `snaps/<user_id>/<uuid>.<ext>`, **4:5 portrait** (`TARGET_RATIO = 4/5`, both modal preview and viewer use `aspect-[4/5]`), **caption max 50 words** (server splits on whitespace + slices). Visibility: `'all_friends'` (no audience rows; feed joins through current `Friend` table — late-added friends can view) or `'selected'` (per-friend `SnapAudience` rows as allowlist). `SnapAudience` doubles as the view-tracker, lazy-created on first view for `all_friends`. `expires_at` = midnight of next day in the **user's IANA TZ** (browser sends `timezone` field; backend uses `zoneinfo`). `SnapUploadView` short-circuits with **403 `{detail:'restricted', restriction_type, expires_at, offense_count}`** when the caller has an active `FunctionRestriction` of `snap_posting` or `both` — `SnapCaptureModal` GETs `/api/restrictions/my/` on mount and never starts the camera if a restriction is in force; a reactive 403 from `POST /api/snaps/` swaps the body to the same banner. `SnapFeedView` and `SnapViewView` also exclude uploaders the caller has a `UserBlock` with (either direction). `SnapViewerModal` footer's `report` button (was a stub) now opens the shared `ReportModal` with `contentType='snap'` + the current snap id. Schedule endpoint (`home()`) emits `id` (Course PK) per entry so the frontend can key snaps to cards.

**Friend page (`/friend`):** `friend/friend.jsx` — receives `allClasses` from `App.jsx` so it can derive "live now" friends (any class today where `nowMins` ∈ [start, end] and the owner isn't `Me`/`currentUser.username`). Header: mono `<N> FRIENDS · <M> IN CLASS NOW` + `ur ppl ◆` Bricolage title + inline search input (debounced 250ms) + dark `+ find more` button (scrolls to suggested panel). Filter pills with counts: `all · N` / `my classes · N` / `my year · N` / `live · N` (replaces the old `nearby` since there's no geo data) + sort dropdown (recently active / live first / name). **While the search input is empty** the page renders a 1fr/320px split: left = horizontal **LIVE NOW** dark strip (up to 5 friend tiles, each `avatar+username+course`) + `recently active` 3-col grid of `FriendCard`s (big avatar with coral ring if live, `LIVE NOW`/`RECENTLY` badge, course pills colored by `dotForCourse(courseId)`, `snapped Xm ago` from `last_snap_at` or `active Xm ago` from `last_seen`, dark `message` button (disabled — "messages coming soon") + cream calendar button). Right rail = `RequestsCard` (inline ✓/× accept-reject) + dark `SuggestedCard` with lilac `Blob` (calls `searchfriends(currentUser.major)` on mount). **Typing in search** swaps the whole grid for the existing compact `SearchResultRow` list. Incoming requests render only in the right-rail RequestsCard, not inline above results.

**Notifications:** `GET /api/notifications/` — fully computed, no stored model. Returns five keys: `friend_requests` (pending `Friend` rows where current user is recipient), `new_snaps` (active non-expired friend snaps with no `SnapAudience has_viewed=True` row for me), `live_class_alerts` (courses live right now — HH:MM string comparison matching frontend `isLiveNow` — where ≥3 friends share the same course; returns `{course_id, course_name, friend_count, friends[]}` capped at 5 friends), `reports_received` (reports against me with `report_document`, `recommended_action`, `status`, `appeal_deadline`, `can_appeal` — limited to the latest 10, excluding terminal statuses), `reports_filed` (my own reports + latest AI doc, also limited to 10). The moderation summary blocks are wrapped in a `try/except` and silently omitted if the moderation tables don't exist, so pre-Phase-7 deploys still serve a payload. **`NotificationsPanel.jsx`** is the single panel component used by both surfaces via a `variant` prop: `variant="dropdown"` (default) — desktop popover anchored below the bell with the absolute-positioned `top-full mt-2 w-[22rem]` container; `variant="fullscreen"` — mobile sheet that fills its parent (`w-full h-full flex flex-col`) with a header × close button. Both variants share the same five sections, the same accept/decline pipeline (`onRespondToRequest` threaded from `App.jsx` → `AppShell` → header/top-bar), the same empty state `all clear ✓`, the same coral bell badge counting `friend_requests.length + new_snaps.length + reports_received.filter(r => r.can_appeal).length` (live alerts + read-only filed-reports excluded — ephemeral or passive), and the same `onRefresh` prop the inline `AppealModal` calls on successful submit to drop the row out of `can_appeal` and clear the badge. The mobile shell is owned by `MobileTopBar.jsx`, which renders a backdrop overlay, locks `document.body.style.overflow` while open, and stops click propagation on the panel itself so background taps close it.

**Models:** `Course` (slots, assignments, exams), `FriendRequest`/`Friendship`, Custom `User` (university, major, grad_year, `last_seen` — nullable indexed datetime, stamped by `LastSeenMiddleware` at most once per 60s on authenticated requests), `Snap` (uploader/course FK, media_file, media_type, caption, visibility, is_removed, expires_at) + `SnapAudience` (snap/viewer FK, has_viewed, viewed_at — unique_together), `ChatRoom` (`room_type` `dm|group` default `dm`, `name` nullable, `created_by`, `linked_snap` nullable + unused in v1 UI, `is_active`), `ChatRoomMember` (`room`/`user` unique_together, `is_admin`, `last_read_at`), `Message` (`room`/`sender`, `content`, `reply_to` nullable, `is_removed`, `removed_at`, `created_at` — default ordering newest-first, index on `(room, -created_at)`). Soft-deleted messages keep their `content` row for future moderation; `MessageSerializer` blanks it client-side via `get_content`. **Moderation models** (migration `0014_moderation`): `Report` (reporter / reported_user FKs, `content_type` `snap|chat_message`, nullable snap/chat_message FKs, `template_reasons` JSON, `free_text`, `status` enum [`pending`, `ai_reported`, `appeal_pending`, `appeal_analyzed`, `third_loop`, `enforced`, `appeal_upheld`, `warned`, `admin_removed`, `admin_dismissed`], `appeal_deadline`), `AiReport` (report / version 1/2/3 unique_together, `provider`, `raw_response` JSON, `report_document` shown to both parties, `violation_likelihood`, `violation_categories` JSON, `recommended_action` `remove|warn|dismiss`), `SimilarityCheck` (one-to-one with Report, `ai_report_v1`/`v2` FKs, cosine `similarity_score`, `provider='text-embedding-3-small'`), `Appeal` (one-to-one with Report, `reason`, `status` `pending|enforced|upheld`), `FunctionRestriction` (`user`, `restriction_type` `snap_posting|chat_messaging|both`, `report` FK nullable, `offense_count`, `expires_at` nullable = permanent, `is_active`), `UserBlock` (`blocker`/`blocked` unique_together, `reason` `appeal_auto|manual`, `report` FK nullable).

**Chat API:** Six endpoints under `/api/chats/`. `GET /` — my DM list (sorted unread → recent → room age) with flattened `other_user`, `last_message`, and `unread_count`. `POST /` `{friend_id}` — create-or-get DM, **friend-gated** (`_are_friends` helper requires `Friend.status=ACCEPTED` either direction; returns 403 `{detail:'not_friends'}` otherwise), deduped via `_find_existing_dm`. `GET /<id>/` — room + last 50 messages, descending. `GET /<id>/messages/?before=<msg_id>&limit=50` — paginated older messages. `POST /<id>/messages/` — text only, server trims, empty→400, >2000 chars→400. `DELETE /<id>/messages/<msg_id>/` — sender-only soft delete. `POST /<id>/read/` — bumps caller's `last_read_at`. `GET /unread/` — `{total: N}` summed across DMs (used by `App.jsx`'s 30s poll). **DM-only in v1**: `room_type='group'` enum value is reserved for a later phase; no group endpoints, no admin flags surfaced. `UserBlock` enforcement is now wired: `ChatListCreateView.post` rejects DM creation with 403 `{detail:'blocked'}`, `ChatListCreateView.get` drops rooms whose only-other peer is blocked in either direction, `MessageListCreateView.post` rejects sends with 403 `{detail:'blocked'}`, and `MessageListCreateView.post` also short-circuits with 403 `{detail:'restricted', restriction_type, expires_at, offense_count}` when a `FunctionRestriction` of `chat_messaging` or `both` is active. `ChatThread.jsx` reads `/api/restrictions/my/` on mount and swaps the input bar for a `ChatRestrictionBanner` (lock icon + expiry copy) when restricted or blocked.
**Moderation API:** Eight user-facing + three admin/cron endpoints. User: `POST /api/reports/` (body `{content_type, snap|chat_message, template_reasons[], free_text}`; dedups same-reporter+same-content in-flight reports → 409, self-reports → 400, missing target → 400), `GET /api/reports/{my,received}/` (prefetches `ai_reports` + `appeal`; `received` is the reported user's view, includes `can_appeal` derived from `status='ai_reported'` + `appeal_deadline > now` + no existing Appeal), `POST /api/appeals/` (reported-user only, `status` must be `ai_reported`, inside deadline, no prior appeal → moves report to `appeal_pending`), `GET /api/appeals/my/`, `GET /api/blocks/` + `DELETE /api/blocks/<id>/` (manual unblock — one-directional; `appeal_auto` rows on both sides have to be lifted separately by each user), `GET /api/restrictions/my/` (used by frontend to pre-disable capture / chat without waiting for 403). Admin/staff: `GET /api/admin/reports/`, `POST /api/admin/reports/<id>/act/` body `{action:'remove'|'dismiss'}`. Cron: `POST /api/admin/run-moderation/` — `permissions.AllowAny` but rejects unless `X-Moderation-Secret` header matches `settings.MODERATION_RUN_SECRET` (no setting → 503). Both `AdminRunModerationView` and the `python manage.py run_moderation` management command call the same `run_moderation_tick()` so Cloud Scheduler can hit either surface.

**Moderation pipeline:** `main/moderation_pipeline.py` is the single source of truth — `run_moderation_tick()` processes reports in three states per pass: `pending` → v1 AI report + 7-day `appeal_deadline` + emails to both parties → `ai_reported`; `appeal_pending` (set when the reported user files an appeal) → v2 + `text-embedding-3-small` cosine similarity → ≥0.60 routes to `enforce()` / `mark_warned()` / `appeal_upheld()` based on v1+v2 majority; <0.60 advances to `third_loop` state (crash-recovery checkpoint), runs v3 tiebreaker, then 2-of-3 remove votes → `enforce()`, all-`warn` → `mark_warned()`, else → `appeal_upheld()`. `extract_content()` handles snap photo (GCS download → base64), snap video (first frame via `cv2.VideoCapture`), and chat message (DB content, even when `is_removed=True`); `call_gpt4o_mini()` uses `response_format={'type':'json_object'}` against the `OPENAI_API_KEY` client. `enforce()` soft-deletes the content, looks up the user's all-time `FunctionRestriction.objects.filter(user=...).count() + 1` as the offense number, applies the duration ladder (3d / 7d / 30d / permanent), and sends an admin alert at offense ≥3. `appeal_upheld()` creates two mutual `UserBlock` rows tagged with the report. `admin_remove` / `admin_dismiss` reuse the same primitives so `POST /api/admin/reports/<id>/act/` and the AI tick produce consistent restriction/email state. The tick also sweeps expired snaps (`is_removed=True` where `expires_at<now`) and flips `is_active=False` on expired restrictions (Phase 9 cleanup folded into the same job — no separate cron).

**Moderation emails:** `main/moderation_email.py` — 7 scenarios via the same Anymail/Resend backend as the rest of the app. `MODERATION_FROM_EMAIL` overrides `DEFAULT_FROM_EMAIL` for these specifically (prod should set `report@timetify.net`); `MODERATION_ADMIN_EMAIL` receives repeat-offender alerts (falls back to from-email). Each send is wrapped in `try/except` so a Resend hiccup never blocks state transitions — the Report row is authoritative.

**Moderation frontend:** `frontend/src/components/shared/ReportModal.jsx` is the shared modal (dark `T.ink` shell, chip rows differ by `contentType` — snap reasons: inappropriate/harassment/spam/violent/hate_speech/other; chat reasons: harassment/hate_speech/threats/spam/private_info/other — plus an optional 1000-char free-text box). 409 `duplicate` is treated as a success no-op so reporters don't get scolded for re-filing. Entry points: `SnapViewerModal` footer's `report` button (was a disabled stub) and the per-bubble ⋮ menu in `ChatThread.jsx` (hover-revealed on desktop, long-press / right-click on touch, also opens via `onContextMenu`). The notification panel surfaces filed-report status + AI doc previews, and an `AppealModal` inline submits `POST /api/appeals/` and calls `onRefresh` to drop the appealable row. `/profile` has a `blocked users` section listing `/api/blocks/` rows with per-row unblock; copy clarifies that `appeal_auto` rows are one-directional. **New icon paths** in `brand.jsx`: `flag`, `dots`, `block` — used by ReportModal, the bubble ⋮ button, and the restriction banners.

**Storage:** Media via `django-storages[google]`. `STORAGES["default"]` toggles to `GoogleCloudStorage` when `GS_BUCKET_NAME` env is set (public URLs, `querystring_auth=False`, IAM-based ACL — bucket should grant `allUsers:objectViewer` for read). Unset locally → falls back to `MEDIA_ROOT`. Cloud Run FS is ephemeral; **prod must have `GS_BUCKET_NAME`**.
**Deploy:** Dockerfile `CMD` runs `python manage.py migrate --noinput && exec gunicorn …` so each new revision self-applies migrations on cold-start. `exec` matters — signals must reach gunicorn.
**External:** OpenAI (GPT-4o-mini for course parsing + AI moderation, `text-embedding-3-small` for appeal-similarity, `OPENAI_API_KEY`), Resend (email), Sentry (errors), Cloud SQL (Postgres), Cloud Run (deploy), AdSense (`ca-pub-9825491172037028`, push in `useEffect`, needs `ads.txt`). **Moderation env vars** (prod-only): `MODERATION_RUN_SECRET` (required to enable `POST /api/admin/run-moderation/`; missing → endpoint returns 503), `MODERATION_FROM_EMAIL` (optional; falls back to `DEFAULT_FROM_EMAIL`), `MODERATION_ADMIN_EMAIL` (optional; receives repeat-offender alerts). ## Security

**Prod settings:** `SECRET_KEY` required (no `django-insecure-`), `SECURE_*`/HSTS/`SECURE_PROXY_SSL_HEADER` auto-enabled. `CORS_ALLOWED_ORIGINS` & `CSRF_TRUSTED_ORIGINS` are explicit allowlists; `ALLOWED_HOSTS` doesn't auto-expand.
**API:** `UserSerializer` read-only: `['id', 'username', 'email', 'status', 'shared_courses', 'last_seen', 'last_snap_at']` — don't loosen `username`/`email` without email verification (`major`/`grad_year` are writable via PATCH). `shared_courses` returns `course_id` strings the caller and target both have (computed per request). `last_snap_at` is a `SerializerMethodField` that reads a precomputed `last_snap_by_user_id` map from context (single `Max('created_at')` aggregate in `FriendListView.get_serializer_context`) — avoids N+1; returns `None` outside the friends-list path. `SearchFriend` username-only (no email search).
**Logging:** Use `logger = logging.getLogger(__name__)`, never `print()` — `send_default_pii=True` leaks data to Sentry.
